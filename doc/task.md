# Описание задачи

## Соответствие датасета требованиям задания

Задание 1 предъявляет четыре обязательных требования к набору данных:

| Требование | Выполнено | Подтверждение |
|------------|-----------|---------------|
| Формат `.csv`, `.xls` или `.xlsx` | Да | Датасет распространяется в формате `.csv` (`starbucks_customer_ordering_patterns.csv`) |
| Одна временная переменная | Да | Колонка `order_date` (тип `datetime`) — используется для сортировки и нарезки на батчи |
| Не менее 10 000 строк | Да | 100 000 строк — в 10 раз больше минимума |
| Не менее 10 признаков, из которых >= 2 категориальных | Да | 20 признаков; категориальные: `drink_category`, `order_channel`, `payment_method`, `loyalty_card_status`, `store_location` и др. |
| Пропущенные значения | Да | Инжектируются программно при инициализации: 5% пропусков в колонки `fulfillment_time_min`, `cart_size`, `num_customizations`, `order_channel`, `drink_category` (`configs/data.yaml`, секция `missing_injection`) |

---

## Данные

**Источник:** [Starbucks Customer Ordering Patterns](https://www.kaggle.com/datasets/likithagedipudi/starbucks-customer-ordering-patterns) (Kaggle)

**Размер:** 100 000 строк, 20 признаков

**Временная переменная:** `order_date` — используется для сортировки и нарезки на батчи

**Целевая переменная:** `total_spend` (непрерывная, в долларах) — задача регрессии

**Ключевые признаки:**

| Признак | Тип | Описание |
|---------|-----|----------|
| `drink_category` | категориальный | тип напитка |
| `order_channel` | категориальный | канал заказа (App, In-store, Drive-thru) |
| `cart_size` | числовой | количество позиций в заказе |
| `num_customizations` | числовой | число кастомизаций |
| `fulfillment_time_min` | числовой | время выполнения заказа (мин) |
| `is_rewards_member` | бинарный | участник программы лояльности |
| `order_time` | временной | время заказа (преобразуется в `hour`) |

При инициализации в 5 колонок инжектируется 5% пропусков для эмуляции реальных данных.

---

## Постановка задачи

Система эмулирует работу ML-конвейера на потоковых данных:

1. Датасет сортируется по `order_date` и нарезается на 10 батчей по 10 000 строк
2. Каждый запуск `update` берёт следующий батч и дообучает модель
3. Цель — предсказывать `total_spend` для новых заказов (`inference`)

---

## Data Quality

Реализовано в `runtime/analysis.py`. На каждом батче вычисляются:

- **% пропусков** по каждой колонке (порог: 50% -> колонка удаляется)
- **% дублей** в батче
- **Выбросы** по z-score (|z| > 3)
- **Data Drift** — KS-тест между текущим и предыдущим батчем; при p < 0.05 выдаётся предупреждение в лог

Результат каждой проверки сохраняется в `artifacts/data/quality/dq_batch_NNN.json`.

## Feature Engineering и подготовка данных

Реализовано в `runtime/preparation.py`:

- `order_time` (строка "HH:MM") -> числовой признак `hour`
- `order_date` -> `month`, `year`
- Удаление ID-колонок (`customer_id`, `order_id`) и временных колонок
- Statsmodels OLS: незначимые числовые фичи (p-value > 0.05) исключаются
- Набор признаков фиксируется на нулевом батче и не меняется (совместимость инкрементальной модели)
- Пропуски: медиана для числовых, "Unknown" для категориальных
- CatBoost работает с категориальными признаками нативно — OHE не применяется

## Проектирование модели

**Алгоритм:** CatBoostRegressor

**Обоснование выбора:**
- Нативная поддержка категориальных признаков без OHE
- Поддержка инкрементального обучения через `init_model`
- Устойчивость к выбросам и пропускам
- Высокое качество из коробки без сложного тюнинга

**Гиперпараметры (configs/training.yaml):**

| Параметр | Значение | Описание |
|----------|----------|----------|
| `iterations_per_batch` | 100 | деревьев на один батч |
| `depth` | 6 | глубина деревьев |
| `learning_rate` | 0.1 | шаг градиентного спуска |
| `loss_function` | RMSE | функция потерь |
| `bootstrap_type` | Bernoulli | тип бутстрапа |
| `subsample` | 0.8 | доля объектов на дерево |

**Инкрементальное обучение:**

```
батч 0  ->  обучение с нуля        ->  100 деревьев
батч 1  ->  fit(init_model=пред.)  ->  200 деревьев
батч N  ->  fit(init_model=пред.)  ->  (N+1) x 100 деревьев
```

---

## Валидация

Реализовано в `runtime/validation.py`:

- **Стратегия:** hold-out, 20% от накопленных данных
- **Метрики:** RMSE, MAE, R²
- **Промоушн в production:** если RMSE улучшилась на >= 0.01 относительно лучшей предыдущей модели
- **Model Drift:** если последняя RMSE хуже исторического минимума на > 10% -> warning в лог
